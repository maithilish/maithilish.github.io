Wildcard bounds in Java API

The previous tutorial covered wildcard bounds in detail, and this tutorial explorers its use in Java API.

Java API extensively uses wildcard bounds, and it is essential to understand them thoroughly. As an example, we analyze java.util.Stream.map() method which is normally used along with lambda expression as follows:

  
List<String> list = Stream.of("USA", "Japan", "India")
                        .map(s -> s.toLowerCase())
                        .collect(Collectors.toList());

At first glance, its method signature is bit scarry: public <R> Stream<R> map(Function<? super T, ? extends R> mapper) which means:

  • it expects a Function which accepts input of type T and returns an object of type R.
  • it returns a Stream of objects of type R.

But, what does Function<? super T, ? extends R> really signifies? To understand it, we need to see the implications had the method is designed without any wildcards: public <R> Stream<R> map(Function<T, R> mapper)

  
Function<Fruit, Fruit> func = f -> new Fruit();

List<Fruit> fruits = new ArrayList<>();    
Stream<Fruit> fruitsStream = fruits.stream().map(func);

In the above code, the Function takes Fruit as input and returns a new Fruit, which is then used with map method on a stream of Fruit. If map api has not used any wildcards, then map method in above code accepts only the Function<Fruit, Fruit> and nothing else. If we try to pass Function<Food, Apple> or any other combination results in error.

To be more flexible, API uses wildcard bounds. For input there are two options: Function<? extents T, R> and Function<? super T, R>. The first option ? extents T is most flexible but makes the input as a Producer. To execute the lambda expression, JVM has to call the functional method Function.apply(T t), and as we have seen in the previous tutorial, calling the add type methods on Producers is not allowed.

The only option left is ? super T which acts as Consumer, and there is no bar on calling apply(T t) on it. It allows T and all its superclass as input, and we are free to use not only Fruit but also its super class, Food, as input. Following combinations are valid:


// on List<Fruit> we can use 
Function<Fruit, Fruit> func = f -> new Fruit();
Function<Food, Fruit> func = f -> new Fruit();

// conversely, Function<Food, Fruit> can be used on
List<Food> foods;
List<Fruit> fruits;
List<Apple> apples;

Finally, tackle the return type. If map api doesn’t use ? extends R then for Function<Food, Apple> the returned stream has to be Stream<Apple> and nothing else.

  
Function<Fruit, Apple> func = f -> new Apple();
Stream<Apple> newStream = fruits.stream().map(func);    // ok
Stream<Fruit> newStream = fruits.stream().map(func);    // error
Stream<Food> newStream = fruits.stream().map(func);     // error

For return type, again there are two options: Function<? super T, ? extends R> and Function<? super T, ? super R>. The ? extends R results in capture error in Function.apply(T t). As producer and consumer is not applicable for return type, the most flexible type, ? extends R, is chosen. With this, we can use the returned stream of Food, Fruit or Apple as Stream<Food>.

    
Function<Fruit, Apple> func = f -> new Apple();
Stream<Apple> newStream = fruits.stream().map(func);    // ok
Stream<Fruit> newStream = fruits.stream().map(func);    // ok
Stream<Food> newStream = fruits.stream().map(func);     // ok

// conversely
Function<Food, Food> func = f -> new Food();
Stream<Food> s = fruits.stream().map(func);     // ok

Function<Food, Fruit> func = f -> new Fruit();
Stream<Food> s = fruits.stream().map(func);     // ok

Function<Food, Apple> func = f -> new Apple();
Stream<Food> s = fruits.stream().map(func);     // ok
