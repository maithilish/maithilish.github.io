Wildcard bounds in Java API

Java API extensively uses wildcard bounds, and we should learn to tackle them. We start with couple of easy ones from java.util.Collections, and then move on to a difficult one in java.util.Stream.

Take the <T> void fill(List<? super T> list, T obj) method of Collections class. If wildcard bound is not there, then we can fill a Dog instance to list of Dog and nothing else. But with List<? super T>, we can fill a Dog not only to List<Dog> but also to List<Pet>.

The List<? super T> acts as consumer, which allows put (write) operations.


List<Pet> pets = new ArrayList<>();
List<Dog> dogs = new ArrayList<>();

Collections.fill(dogs, new Dog());
Collections.fill(pets, new Dog());

Next, take the Collections.copy(List<? super T> dest, List<? extends T> src) method. If wildcards is not used then we can copy list of a type to another list of same type.


List<Pet> pets = new ArrayList<>();
List<Pet> allPets = new ArrayList<>();

Collections.copy(allPets, pets);

However, with wildcards bounds; if T is Dog, then we can copy list of Dog or Bulldog to list of Dog or Pet. The List<? extends T> src acts as producer that allows only the get (read) group of operations, while List<? super T> dst acts as consumer which allows put (write) operations.


List<Pet> pets = new ArrayList<>();
List<Dog> dogs = new ArrayList<>();
List<Bulldog> bulldogs = new ArrayList<>();
List<Cat> cats = new ArrayList<>();

Collections.copy(pets, pets);
Collections.copy(pets, dogs);
Collections.copy(pets, bulldogs);
Collections.copy(pets, cats);

Collections.copy(dogs, dogs);
Collections.copy(dogs, bulldogs);

The final example is quizzical java.util.Stream.map() method which is normally used with lambda expression as follows:

  
List<String> list = Stream.of("USA", "Japan", "India")
                        .map(s -> s.toLowerCase())
                        .collect(Collectors.toList());

At first glance, its map() method signature is quite cryptic.


public <R> Stream<R> map(Function<? super T, ? extends R> mapper)

which broadly means:

  • map input parameter is a Function that accepts input of type T and returns an object of type R.
  • map returns a Stream of objects of type R.
  • R is the type parameter defined at method level, so function can return instance of any type.

But, what does Function<? super T, ? extends R> really signifies? To know that, we need to understand the implications had it been designed without any wildcards as public <R> Stream<R> map(Function<T, R> mapper)

List<Dog> dogs = new ArrayList<>();    

Function<Dog, Cat> fn = e -> new Cat();

dogs.stream().map(fn);

The above code, defines a Function that takes Dog as input and returns a new Cat and use it to map a stream of Dog. If map api is without any wildcards then it accepts only the Function with input as Dog and nothing else. If we try to pass Function<Pet, Cat> or any other input type results in compiler error. Return type R can be any type.

To be more flexible, API uses wildcard bounds. For input, there are two options: Function<? extends T, R> or Function<? super T, R>. The first option ? extents T is most flexible, but it makes the input as a Producer. To execute the lambda expression, map() calls the functional method Function.apply(T t), which is a put method as it parameter T is a type parameter. As we have seen in the previous tutorial, put (write) methods are not allowed on Producers. So, extends is ruled out.

The other option, ? super T acts as Consumer, and there is no bar on calling apply(T t) on it. It allows T and all its supertype as input. Now, we are free to use not only Dog but also its super class, Pet, as input. Following combinations are valid:


List<Pet> pets = new ArrayList<>();
List<Dog> dogs = new ArrayList<>();
List<Bulldog> bulldogs = new ArrayList<>();

Function<Pet, String> fn1 = e -> "foo";
Function<Dog, String> fn2 = e -> "bar";
Function<Bulldog, String> fn3 = e -> "zoo";

pets.stream().map(fn1);     // fn2 and f3 - compiler error

dogs.stream().map(fn1);     // f3 - compiler error
dogs.stream().map(fn2);
        
bulldogs.stream().map(fn1);
bulldogs.stream().map(fn2);
bulldogs.stream().map(fn3);

// we can use 
// Function<Pet, String> on list of Pet, Dog and Bulldog
// Function<Dog, String> on list of Pet and Dog

Next, let’s tackle the function return type if function is specified as Function<? super T, R>

  
Function<Pet, Pet> fn1 = e -> new Bulldog();
Function<Pet, Dog> fn2 = e -> new Bulldog();
Function<Pet, Bulldog> fn3 = e -> new Bulldog();

Stream<Pet> s1 = pets.stream.map(fn1);   // allowed: only fn1
Stream<Dog> s2 = pets.stream.map(fn2);   // allowed: only fn2
Stream<Bulldog> s3 = pets.stream.map(fn3);   // allowed: only fn3

When map is assigned to Stream<Pet>, R is set to Pet; only fn1 is allowed and we can neither use fn2 nor fn3. Similarly, for Stream<Dog> we can use a Function with return type Dog and nothing else.

To make things more flexible, API uses wildcard bounds. For return type, again there are two options: ? extends R and ? super R. As producer and consumer is not applicable for return type, the most flexible type, ? extends R, is chosen.


Function<Pet, Pet> fn1 = e -> new Cat();
Function<Pet, Dog> fn2 = e -> new Dog();
Function<Pet, Bulldog> fn3 = e -> new Bulldog();

Stream<Pet> s1 = pets.stream().map(fn3); // allowed: fn1, fn2, fn3
Stream<Dog> s2 = pets.stream().map(fn3); // allowed: fn2, fn3
Stream<Bulldog> s3 = pets.stream().map(fn3); // allowed: fn3

In the first map statement, R is Pet so then functions (fn1, fn2 and f3) that returns Pet or its subtypes are allowed. In the second map call, R is Dog and functions (f3 and f3) that returns Dog or Bulldog is allowed.
